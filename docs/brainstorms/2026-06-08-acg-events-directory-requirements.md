# ACG 活动目录需求文档

## 摘要

构建首个公开版本的中国 ACG 活动半可信目录，面向想快速发现活动的普通 ACG 爱好者。核心闭环是发现：筛选附近和近期活动，打开活动详情页，并在有官方 QQ 群或购票渠道时继续跳转到官方渠道。

---

## 问题背景

想参加中国 ACG 活动的普通爱好者，通常需要关注大量主办方、同好会和小圈层传播渠道，才能知道个别活动。中国有大量民间活动，但发现路径足够分散，以至于想参加的人没有清晰的起点。

产品应该降低发现活动的成本，但不变成另一个信息流或社区。它的价值来自收集活动事实信息、发布前审核，以及让公开目录足够容易扫读。

---

## 关键决策

- **半可信目录，而不是社区平台** - 网站聚焦已审核的活动信息和官方外部渠道，不做评论、关注、私信或站内讨论。
- **低门槛提交，审核后公开** - 用户可以提交活动信息，联系方式可选；任何内容都必须经过维护者审核后才公开。
- **用户字符串在审核时转为正式词条** - 提交者把活动类型和活动 IP 作为自由文本填写；维护者在批准前将其映射到正式目录词条，或创建新词条。
- **规模辅助列表排序，热度辅助首页排行** - 普通列表优先使用维护者管理的活动规模排序，首页热门榜使用去重后的详情页访问量。
- **封面图可选** - 提交者可以附带一张封面图，图片随活动一起审核，通过后才公开展示。
- **v1 提供完整维护后台** - 维护后台覆盖审核，以及活动、规模标签、活动类型、活动 IP 词条、封面图和历史记录的持续管理。

---

## 参与者

- A1. **访客** - 想查找附近或近期可参加活动的普通 ACG 爱好者。
- A2. **提交者** - 无需账号即可贡献活动信息的访客或组织者。
- A3. **维护者** - 审核提交、规范目录数据并管理公开记录的可信运营者。

---

## 关键流程

- F1. 活动发现
  - **触发：** 访客想找到近期值得参加的活动。
  - **参与者：** A1
  - **步骤：** 访客打开首页或浏览页，按地点、时间、类型、活动 IP 或规模筛选，扫读排序后的结果，并打开活动详情页。
  - **结果：** 访客获得足够可信的活动信息，可以决定是否继续前往官方渠道。
  - **覆盖：** R1, R2, R3, R4, R5, R6

- F2. 活动提交
  - **触发：** 提交者知道一个目录中还没有的活动。
  - **参与者：** A2
  - **步骤：** 提交者填写必需的活动事实信息，可选填写联系方式并上传一张封面图，然后提交审核。
  - **结果：** 提交进入维护者队列，在批准前不会出现在公开发现路径中。
  - **覆盖：** R10, R11, R12, R13, R14

- F3. 维护者审核与规范化
  - **触发：** 待审核提交需要被接受、拒绝或修正。
  - **参与者：** A3
  - **步骤：** 维护者检查提交的事实信息，将自由文本的类型和活动 IP 映射到正式词条，分配规模标签，审核封面图，并批准或拒绝提交。
  - **结果：** 批准后的活动成为公开目录记录；拒绝的活动保持隐藏，并保留内部审核轨迹。
  - **覆盖：** R15, R16, R17, R18, R19

- F4. 热门活动浏览
  - **触发：** 访客想快速了解近期受关注的未结束活动。
  - **参与者：** A1
  - **步骤：** 访客在首页切换近 3 日、7 日和 30 日热门榜，并打开榜单中的活动。
  - **结果：** 首页突出展示近期有独立访客关注的未结束活动。
  - **覆盖：** R20, R21, R22

---

## 需求

**公开发现**

- R1. 公开目录必须支持访客按地点、时间、活动类型、活动 IP 和活动规模筛选活动。
- R2. 默认公开发现视图必须聚焦尚未结束的活动。
- R3. 普通活动列表必须优先按照维护者管理的活动规模排序，再使用近期程度或日期等次级排序。
- R4. 活动卡片必须提供足够扫读的信息：名称、地点、日期范围、类型、活动 IP、规模、有封面图时展示封面图，以及进入详情页的路径。
- R5. 筛选结果状态必须可以分享或恢复，让访客能回到同一发现上下文。
- R6. 空状态必须说明没有找到匹配的近期活动，但不能暗示目录里完全没有活动。

**活动详情**

- R7. 公开活动详情页必须展示活动名称、地点、日期范围、类型、活动 IP、规模、可用时的官方 QQ 群、可用时的购票地址，以及可用时的封面图。
- R8. 已结束活动必须保留公开详情页可访问，但不能出现在默认近期列表或首页热门榜中。
- R9. 当官方外部渠道存在时，详情页必须让访客容易到达这些渠道。

**提交**

- R10. 提交者必须能在不创建账号的情况下提交活动。
- R11. 活动名称、地点、开始时间、结束时间、活动类型文本和活动 IP 文本在提交时为必填。
- R12. 官方 QQ 群、购票地址、提交者联系方式和一张封面图在提交时为可选。
- R13. 提交成功后必须告知提交者活动正在等待审核，v1 不提供提交者状态页。
- R14. 提交记录和封面图必须在维护者批准前保持隐藏，不能进入公开发现路径。

**维护者流程**

- R15. 维护者必须能审核待处理提交、编辑活动事实信息、批准记录、拒绝记录，并保留审核历史。
- R16. 维护者必须能管理全部活动记录，包括已发布、待审核、已拒绝、已归档和历史活动。
- R17. 维护者必须能创建、重命名、排序、停用和分配活动规模标签。
- R18. 维护者必须能管理活动类型和活动 IP 词表，并在审核时把提交字符串映射到正式词条。
- R19. 维护者必须能在发布前后审核、替换或移除活动封面图。

**热度**

- R20. 每个活动必须有基于详情页访问的热度值，同一访客 IP 在同一活动的同一统计窗口内只计一次。
- R21. 首页必须展示尚未结束活动的近 3 日、近 7 日和近 30 日热门榜。
- R22. 热门榜必须链接到活动详情页，并且不能包含待审核、已拒绝、已归档或已结束活动。

**信任、反滥用和隐私**

- R23. 公开访客不得看到未批准提交、已拒绝记录、内部审核备注、提交者联系方式或原始访客 IP 数据。
- R24. 提交入口应该具备适合低门槛公开表单的基础反滥用保护。
- R25. 热度统计必须保留“同 IP 只记 1 次”的产品规则，同时不能在面向公开记录中存储明文访客 IP。

## 验收示例

- AE1. **覆盖 R1, R2, R4。** 给定上海和广州都有已批准的近期活动，当访客按上海筛选时，只显示匹配的已批准近期上海活动。
- AE2. **覆盖 R3, R17。** 给定两个匹配的近期活动拥有不同的规模标签，当访客查看普通列表时，规模标签优先级更高的活动排在前面。
- AE3. **覆盖 R7, R9。** 给定一个已批准活动有官方 QQ 群和购票地址，当访客打开详情页时，两个官方渠道都可见且可到达。
- AE4. **覆盖 R8, R21, R22。** 给定某活动昨天已经结束，当首页热门榜渲染时，即使该活动详情页仍可通过直达链接访问，它也不会出现在热门榜中。
- AE5. **覆盖 R10, R11, R12, R13, R14。** 给定提交者填写必填字段并省略可选联系方式和封面图，当他们提交活动时，提交成功并进入待审核状态。
- AE6. **覆盖 R15, R18。** 给定一个待审核提交包含新的活动 IP 字符串，当维护者审核它时，维护者可以先创建或选择正式活动 IP 词条，再批准活动。
- AE7. **覆盖 R20。** 给定同一访客 IP 在一个热度窗口内多次打开同一活动详情页，当计算热度时，该访客在该窗口内只贡献一次计数。
- AE8. **覆盖 R23。** 给定一个待审核提交包含提交者联系方式，当公开访客搜索目录时，该提交和联系方式都不可见。

---

## 成功标准

- 访客可以完成发现闭环：筛选附近近期活动，打开详情页，并在官方渠道存在时继续前往官方渠道。
- 维护者可以保持公开数据一致，而不要求提交者理解网站的正式类型、IP 或规模词表。
- 首页可以通过近 3 日、7 日和 30 日热门榜传达近期关注度，而不把热度变成商业 KPI 仪表盘。
- 网站可以接收有用的社区提交，同时把未审核内容挡在公开目录之外。

## 范围边界

### 后续再做

- 提交者状态页、查询码和自助修改。
- 超出 `STRATEGY.md` 中轻量健康信号的高级分析。
- 面向已结束活动的专门历史浏览页；v1 只保留直达详情页可访问。

### 不属于产品定位

- 站内社交功能，例如评论、关注、收藏、私信或社区信息流。
- 独立售票或支付流程；网站只指向官方外部渠道。
- 完全匿名且无需审核、提交后立即公开的发布板。

---

## 依赖和假设

- 项目继续遵循 `STRATEGY.md` 的方向：半可信目录、普通 ACG 爱好者为主要用户、不做站内社交层。
- 首版允许把 Cloudflare 生态作为托管、存储、反滥用和维护者访问控制的运行约束。
- 有维护者可以审核提交并管理正式词表；产品不试图完全依靠自动化解决信任问题。
- 活动规模标签、活动类型词条和活动 IP 词条都是运行时管理的目录数据，而不是写死在产品里的固定文案。
