# 管理员增加活动

## Goal

让已通过后台身份验证的管理员可以在管理系统内直接录入活动，无需通过面向普通用户的 Turnstile 投稿流程。管理员录入的活动必须使用标签库中的规范标签，同时允许管理员在录入过程中创建新的规范标签。

## Background

- 公开投稿页面为 `/submit`，提交到 `POST /api/submit`，并经 Turnstile 校验后写入 `status = 'pending'` 的活动。
- 公开投稿只保存自由文本 `tag_suggestions`，不得创建 `tags` 或 `event_tags` 记录。
- 管理员编辑页面 `/admin/events/:id/edit` 已加载全部规范标签，并通过 `TagInput.svelte` 选择已有标签或输入新标签。
- `editEvent()` 已实现标签复用、新标签创建、别名解析和 `event_tags` 关联替换，但当前没有管理员新增活动的页面或 API。
- 管理员 API 位于 `/api/admin/**`，由现有中间件统一执行管理员身份验证，并返回统一 JSON 封装。

## Requirements

- R1. 在管理后台提供明确可发现的“增加活动”入口和受保护页面。
- R2. 新增表单覆盖现有活动的可编辑字段：名称、类型、规模、行政区、场馆、详细地址、开始/结束日期与时间、封面、描述、QQ 群、购票地址、来源链接和联系信息。
- R3. 标签控件加载标签库内所有未归并的规范标签，支持选择一个或多个已有标签。
- R4. 管理员可以在同一标签控件中输入新标签；保存时新标签进入标签库并与新活动关联。
- R5. 新增活动必须至少关联一个规范标签，并遵守现有标签名称长度、去重、大小写不敏感和别名解析约束。
- R6. 后端复用或提取现有管理员表单校验，验证日期/时间顺序、固定类型/规模、行政区代码、URL 和必填字段，不仅依赖浏览器校验或 D1 约束。
- R7. 提供受保护的管理员创建 API，成功时返回新活动 ID，失败时使用现有 `{ ok, error }` JSON 协议展示可操作错误。
- R8. 活动、新标签和标签关联必须作为一个原子操作成功或失败，不留下半成品活动。
- R9. 创建成功后写入管理审计日志，并把管理员导向可继续编辑或查看新活动的页面。
- R10. 新增页面遵守现有管后台紧凑、操作导向的响应式视觉与无障碍约定，并复用现有 Astro/Svelte/Bits UI 控件。
- R11. 普通用户投稿行为保持不变：仍需 Turnstile，仍进入待审核，仍只保存标签建议。
- R12. 管理员新增的活动直接写入 `published` 状态，并设置 `published_at`，创建后可立即在公开列表中出现。

## Acceptance Criteria

- [ ] AC1. 管理员可从桌面和移动后台导航进入新增活动页面；未认证请求仍被现有管理员鉴权拦截。
- [ ] AC2. 管理员能够填写与现有投稿/编辑约定相容的活动字段，选择已有标签，成功创建活动。
- [ ] AC3. 管理员输入一个不存在的合法标签名后，该标签出现在标签库中并与新活动关联；重复名称不创建额外标签。
- [ ] AC4. 未选择任何标签、超长标签或无效表单数据均不会写入活动，页面会显示后端返回的明确错误。
- [ ] AC5. 任一活动/标签/关联写入失败时，数据库不留下部分新数据。
- [ ] AC6. 成功创建后产生包含新活动 ID 和创建元数据的审计记录，前端给出成功反馈并导向新活动的后续工作流。
- [ ] AC7. 公开投稿的 Turnstile、`pending` 状态、标签建议和 API 合同没有改变。
- [ ] AC8. 项目的格式化、静态类型检查、lint、构建以及针对创建/标签原子性的自动验证通过。

## Out Of Scope

- 修改普通用户投稿页面或让普通用户直接管理规范标签。
- 重做现有活动编辑、审核、下线、重新发布或标签归并流程。
- 新增标签删除、重命名或独立标签创建页面。
- 上传本地封面图片；继续使用封面 URL。

## Decisions

- D1. 根据用户确认，管理员新增活动直接为已发布；不进入待审核队列。
